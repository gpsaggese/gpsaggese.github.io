# 4 / 27: Git Branching

</center>

<center>

![](data605/book/Lesson02.1-Git.png/slides004.png){width=80%}

</center>

- **Branching**
  - _Branching_ in Git allows developers to create a separate line of
    development. This means you can work on new features or bug fixes without
    interfering with the main codebase. It's like creating a parallel universe
    where you can experiment freely.

- **Why branch?**
  - Branching lets you work independently without impacting the main code. This
    is crucial for maintaining stability in the main branch, especially in
    collaborative projects.
  - By branching, you can avoid making changes directly to the main branch,
    which helps prevent accidental errors from affecting the entire project.
  - Once your work on a branch is complete, you can merge your changes back into
    the main branch. This process ensures that updates are systematically
    integrated.
  - Merging code upstream means incorporating your completed work into the main
    branch, ensuring that all team members have access to the latest updates.

- **Git branching is lightweight**
  - Creating a branch in Git is quick and doesn't require much space. It's
    essentially just a pointer to a specific commit in the project's history.
  - Unlike some other version control systems, Git doesn't store changes as file
    differences. Instead, it takes snapshots of the entire project at each
    commit, making branching efficient.

- **Git workflows branch and merge often**
  - In Git, it's common to branch and merge multiple times a day. This might be
    surprising for those used to centralized version control systems, where
    branching is less frequent.
  - Developers might even create a branch for a short task, like before lunch,
    to keep their work organized and isolated.
  - Branches in Git are inexpensive to create, so it's encouraged to use them
    liberally to manage different tasks and features effectively.

# 5 / 27: Git Branching

</center>

<center>

![](data605/book/Lesson02.1-Git.png/slides005.png){width=80%}

- **`master` (or `main`) is a normal branch**
  - In Git, the `master` or `main` branch is the default branch where the main
    line of development occurs. It's essentially a _pointer_ to the latest
    commit in the project. As you make new commits, this pointer moves forward
    to include the latest changes. This is why it's often referred to as the
    "tip" of the branch.

- **`HEAD`**
  - `HEAD` is a special pointer in Git that indicates the current branch you are
    working on. For example, if you are working on the `master` branch, `HEAD`
    points to `master`. When you switch branches using `git checkout <BRANCH>`,
    `HEAD` moves to point to the new branch, allowing you to work on different
    parts of your project without affecting the main line of development.

- **`git branch testing`**
  - This command creates a new branch named `testing`. It acts as a new pointer
    that starts at the current commit where `HEAD` is located. This new branch
    is independent and can be moved forward as you make new commits, allowing
    you to experiment or develop features without affecting the main branch.

- **Divergent history**
  - When you create a new branch and start making changes, the history of
    commits can diverge. This means that the `master` branch and the `testing`
    branch can have different sets of commits, representing different lines of
    development. This is useful for working on features or fixes in isolation
    before merging them back into the main branch.

- **`git checkout` switches branches**
  - The `git checkout` command is used to switch between different branches in a
    Git repository. This is a common operation when you want to work on
    different features or versions of your project.
  - When you use `git checkout`, it moves the _HEAD_ pointer to the branch you
    specify. The _HEAD_ is a reference to the current branch you are working on.
  - It also updates the files in your working directory to match the state of
    the branch you switched to. This means that any changes or files that are
    specific to the new branch will be reflected in your working directory.

- **Example with two branches: `master` and `testing`**
  - Imagine you have two branches in your project: `master` and `testing`.
  - If you are currently on the `master` branch and you want to switch to the
    `testing` branch, you would use the command `git checkout testing`.
  - This action moves the _HEAD_ pointer from `master` to `testing`, and your
    working directory will now reflect the files and changes that are part of
    the `testing` branch.
  - You can continue to work on the `testing` branch, making changes and
    committing them. Each commit will move the pointer of the `testing` branch
    forward, capturing the new state of your work.

- **Need a hotfix** to `master`
  - This section explains how to apply a quick fix, known as a _hotfix_,
    directly to the `master` branch in Git.
  - The process begins by switching to the `master` branch using
    `git checkout master`.
  - A new branch named `hotfix` is created from `master` with
    `git checkout -b hotfix`. This allows you to work on the fix without
    affecting the `master` branch until it's ready.
  - After making the necessary changes, you commit them with
    `git commit -am "Hot fix"`. The `-am` flag is used to add and commit changes
    with a message in one step.
  - Finally, you switch back to the `master` branch and merge the `hotfix`
    branch into it using `git merge hotfix`. This incorporates the hotfix
    changes into `master`.

- **Fast forward**
  - This term refers to a specific type of merge in Git where the `master`
    branch is directly updated to match another branch, like `iss53`, without
    creating a new commit.
  - A fast forward merge happens when there is no divergent history between the
    branches, meaning the `master` branch can simply be moved forward to the
    latest commit of the other branch.
  - This is a clean and simple way to update branches when no conflicts or
    additional history need to be resolved.

# 10 / 27: Fast Forward Merge

</center>

<center>

![](data605/book/Lesson02.1-Git.png/slides010.png){width=80%}

</center>

- **Fast Forward Merge**: This is a type of merge in Git where you combine two
  commits, `Y` and `X`, but `X` is already part of the history of `Y`. This
  means that `Y` is a direct descendant of `X`.

- **No Divergent History**: In this scenario, there are no separate paths or
  branches that need to be reconciled. Since `Y` is directly reachable from `X`,
  Git doesn't need to create a new commit to merge them. Instead, it simply
  updates the branch pointer to point to `Y`.

- **Mental Model**: Think of a branch as a label or pointer that marks the
  latest commit in that branch. When you perform a fast forward merge, you're
  essentially just moving this label forward to the latest commit.

- **Example**: If you have a branch `master` and another branch `experiment`,
  and `experiment` is ahead of `master` without any conflicting changes, you can
  merge them with:

  ```bash
  > git checkout master
  > git merge experiment
  ```

  This command moves the `master` branch pointer to the latest commit on
  `experiment`, effectively making `master` up-to-date with `experiment`.

- **Visual Representation**: The images likely show how the branch pointer moves
  from one commit to another, illustrating the simplicity of a fast forward
  merge. This visual aid helps in understanding that no new commits are created
  during this process.

# 11 / 27: Merging Conflicts

</center>

<center>

![](data605/book/Lesson02.1-Git.png/slides011.png){width=80%}

</center>

- **Merging Conflicts**:
  - This slide is about handling situations where _Git_, a version control
    system, encounters conflicts during a merge. Merging is when you combine
    changes from different branches of a project. Sometimes, Git can't
    automatically merge changes because of conflicts.

- **Tutorial**:
  - A tutorial is provided to help understand merging conflicts. It can be found
    [here](https://github.com/gpsaggese/umd_classes/blob/master/data605/tutorials/tutorial_git/tutorial_git.md#merging-conflicts).
    This resource is useful for learning how to handle these situations
    effectively.

- **When Git Can't Merge**:
  - Conflicts occur when the same file is modified in different ways on two
    branches. For example, if both branches change the same line in a file, Git
    won't know which change to keep.
  - Another common conflict is when one branch modifies a file while another
    branch deletes it. Git can't decide whether to keep the changes or delete
    the file.

- **Git's Behavior During Conflicts**:
  - Git will not automatically create a merge commit when it encounters a
    conflict. Instead, it pauses the process to allow the user to resolve the
    conflict manually.
  - It adds conflict resolution markers in the files to show where the conflicts
    are. These markers help users identify the conflicting changes.

- **Manual Conflict Resolution**:
  - Users need to manually edit the files to resolve conflicts. This can be done
    using tools like `git mergetool`, which helps in comparing and editing the
    conflicting files.
  - After resolving the conflicts, users must use `git add` to mark the
    conflicts as resolved.
  - Finally, a `git commit` is needed to complete the merge process.
  - Integrated development environments (IDEs) like PyCharm or VS Code can
    assist in resolving conflicts by providing visual tools to compare and edit
    files.

<center>

![](data605/book/Lesson02.1-Git.png/slides012.png){width=80%}

</center>

- **Two Ways of Merging Divergent History in Git**
  - In Git, when you have two branches that have diverged from a common point,
    you can bring them together using either a _merge_ or a _rebase_. This slide
    explains these two methods using an example with branches `master` and
    `experiment`, which share a common ancestor `C2`.

- **Merge**
  - To merge, you first switch to the target branch, in this case, `master`.
  - You then execute the `git merge experiment` command. This creates a new
    commit, `C5`, which combines the changes from both branches. This method
    keeps the history of both branches intact, showing a clear path of how the
    branches came together.

- **Rebase**
  - Rebasing involves switching to the branch you want to rebase, here
    `experiment`.
  - The `git rebase master` command is used. This process involves taking all
    the changes made in `experiment` since the common ancestor `C2` and applying
    them on top of `master` at `C3`.
  - This method rewrites the commit history, making it appear as though the
    changes from `experiment` were made after `master`, resulting in a linear
    history. Only the current branch (`experiment`) is affected, and it is
    fast-forwarded to include the changes.

  - The images on the right side of the slide likely illustrate the differences
    in commit history between merging and rebasing, helping to visualize how

# 13 / 27: Uses of Rebase

</center>

<center>

![](data605/book/Lesson02.1-Git.png/slides013.png){width=80%}

</center>

- **Rebasing makes for a cleaner history**
  - When you rebase, it rearranges the commit history to appear as if all
    changes were made one after the other, rather than simultaneously. This
    creates a linear and easy-to-follow history.
  - In reality, multiple developers might be working on different features or
    fixes at the same time, but rebasing makes it look like these changes were
    made sequentially. This can make it easier to understand the progression of
    the project over time.

- **Rebasing to contribute to a project**
  - _Developer_: If you're contributing to a project that you don't own, you
    typically work on a separate branch. Once your work is complete, you rebase
    your branch onto the latest version of the main branch (often called
    `origin/master`). This ensures your changes are up-to-date with the main
    project.
  - _The maintainer_: By rebasing, you make it easier for the project maintainer
    to incorporate your changes. They can simply fast forward the main branch to
    include your work, or apply your changes without dealing with conflicts,
    making the integration process smoother and more efficient.

# 14 / 27: Golden Rule of Rebasing

</center>

<center>

![](data605/book/Lesson02.1-Git.png/slides014.png){width=80%}

</center>

- **Golden Rule of Rebasing**
  - **Remember**: When you rebase, you are essentially taking existing commits
    and creating new ones that are similar but not identical. This means the
    history of your project changes, which can lead to complications if not
    handled carefully.

- **Problem**
  - **You push commits to a remote**: This means you've shared your work with
    others, and they might start using it as a base for their own work.
  - **Others pull commits and base work on them**: Your collaborators download
    your changes and start building their own work on top of it.
  - **You rewrite commits with `git rebase`**: Rebasing changes the commit
    history, which can make the history look cleaner but also changes the commit
    IDs.
  - **You push again with `git push --force`**: This action forces the new
    commit history onto the remote, overwriting the old history.
  - **Collaborators must re-merge work**: Because the commit history has
    changed, your collaborators will have to adjust their work to fit the new
    history, which can be time-consuming and error-prone.

- **Solution**
  - **Strict**: _"Do not ever rebase commits outside your repository"_: This
    means you should avoid rebasing commits that have already been shared with
    others to prevent disrupting their work.
  - **Loose**: _"Rebase your branch if only you use it, even if pushed to a
    server"_: If you are the only one working on a branch, it's generally safe
    to rebase, even if the branch is on a remote server, because no one else is
    affected by the changes.

The images likely illustrate the difference between a clean commit history with
rebasing and a more complex one without it, emphasizing the importance of
understanding when and how to use rebasing effectively.

# 15 / 27: Rebase vs Merge: Philosophical Considerations

</center>

<center>

![](data605/book/Lesson02.1-Git.png/slides015.png){width=80%}

</center>

- **Rebase vs Merge: Philosophical Considerations**
  - The choice between **Rebase** and **Merge** is not just a technical decision
    but also a philosophical one. It revolves around how you perceive the commit
    history of a repository. The key question to ask is: _What does the commit
    history of a repo mean?_

- **History is the record of what actually happened**
  - This perspective values authenticity and transparency. The idea is that the
    commit history should reflect the true sequence of events, even if it
    appears messy or complicated. This approach is akin to keeping a raw,
    unedited diary.
  - If you align with this view, you would typically use `git merge`. Merging
    keeps all the original commits intact, showing exactly how the project
    evolved over time, including all branches and merges.

- **History represents how a project should have been made**
  - This viewpoint prioritizes clarity and readability for future developers. It
    suggests that the commit history should be curated to present the most
    logical and understandable sequence of events, even if it means altering the
    original order.
  - If you prefer this approach, you would use `git rebase` and possibly
    `filter-branch`. Rebasing allows you to rewrite commit history, creating a
    linear and cleaner narrative that can be easier for others to follow and
    understand.

In summary, the decision between rebase and merge is about choosing between
preserving the authentic history of a project or crafting a more streamlined and
comprehensible version for future reference.

# 17 / 27: Remote Branches

</center>

<center>

![](data605/book/Lesson02.1-Git.png/slides017.png){width=80%}

</center>

- **Remote branches** are essentially bookmarks or pointers that represent
  branches in a remote repository. They help you keep track of what the state of
  the branches is on the remote server. When you run the command
  `git remote -v`, it shows you the URLs of the remote repositories you have
  configured, both for fetching and pushing changes. This is crucial for
  collaboration, as it allows multiple developers to work on the same project
  from different locations.

- **Tracking branches** are local branches that have a direct relationship with
  a remote branch. For example, your local `master` branch might track the
  `origin/master` branch on the remote. This means that your local branch is set
  up to pull changes from and push changes to the remote branch. You cannot
  directly modify a remote branch like `origin/master`; instead, you make
  changes to your local tracking branch and then push those changes to the
  remote. When you run `git fetch origin` or `git pull`, Git updates your
  tracking branches to reflect the latest state of the remote branches.

- To share your work with others, you need to push your local branch to the
  remote repository. For instance, if you have a local branch called
  `serverfix`, you can share it by pushing it to the remote with the command
  `git push origin serverfix`. This makes your changes available to others who
  have access to the remote repository.

- If you want to start working on a branch that exists on the remote but not
  locally, you can create a new local branch that tracks the remote branch. The
  command `git checkout -b serverfix origin/serverfix` does exactly this. It
  creates a new local branch called `serverfix` that tracks the `serverfix`
  branch on the remote, allowing you to work on it and later push your changes
  back to the remote.

# 18 / 27: Git Workflows

</center>

<center>

- **Git workflows**: This refers to the different methods or strategies teams
  use to collaborate and manage code changes using Git. Git is a version control
  system that helps track changes in code, making it easier for multiple people
  to work on the same project without conflicts.

- **Long-running branches**: These are branches in a Git repository that remain
  open for extended periods. They serve different purposes and have varying

  - **`master` branch**: This is the main branch that is always ready for
    release. It contains the most stable version of the code.
  - **`develop` branch**: This branch is used for ongoing development. It is
    where new features and changes are integrated before they are stable enough
    to be merged into the `master`.
  - **Topic/feature branches**: These are temporary branches created to work on
    specific features or fixes. Once the work is complete and tested, these
    branches are merged into the `develop` branch.
  - _Merging_: When a branch is considered "stable enough," it is merged into a
    higher-level branch, like `develop` or `master`, to ensure the main codebase
    remains stable and up-to-date.

- **Topic branches** are a key feature in Git that allow developers to create
  _short-lived branches_ for working on a single feature or fix. These branches
  are temporary and are used to isolate work until it is ready to be integrated
  into the main codebase.
  - For example, branches like `hotfix` or `wip-XYZ` are created to address
    specific issues or work-in-progress tasks.
  - These branches are _easy to review_ because they contain changes related to
    a single feature or fix, making it simpler for team members to understand
    and provide feedback.
  - They are _siloed from the rest_ of the codebase, meaning changes in these
    branches do not affect the main code until they are merged. This isolation
    helps prevent unfinished or experimental code from disrupting the stable
    version.
  - Git excels at handling branches, unlike some other version control systems

  - For instance, you might start working on a branch named `iss91`, realize
    some changes need to be discarded, and then continue on a new branch
    `iss91v2`.
  - Sometimes, a branch like `dumbidea` might be created for experimental
    purposes and mistakenly merged into the main branch (`master`), which can
    lead to issues if not managed carefully.
  - When you are satisfied with the changes in your branch, you can use a
    technique called _squash-and-merge_ to combine all the changes from
    `iss91v2` into a single commit before merging it into the main branch. This
    helps keep the commit history clean and organized.

- **Centralized workflow in centralized VCS**
  - _Developers_:
    - **Check out the code from the central repo on their computer**: This means
      that developers download the latest version of the code from a central
      repository to their local machine. This is the starting point for any
      changes they want to make.
    - **Modify the code locally**: Developers work on their own computers to
      make changes to the code. This allows them to test and develop features
      independently.
    - **Push it back to the central hub**: Once changes are complete, developers
      upload their modified code back to the central repository. If someone else
      has changed the code in the meantime, they may need to resolve conflicts
      before their changes can be accepted.

- **Centralized workflow in Git**
  - _Developers_:
    - **Have push (i.e., write) access to the central repo**: This means
      developers can directly upload their changes to the central repository,
      but they need permission to do so.
    - **Need to fetch and then merge**: Before pushing changes, developers must
      first download the latest version of the code (fetch) and integrate any
      new changes into their work (merge). This ensures that their changes are
      compatible with the latest version.
    - **Cannot push code that will overwrite each other's code**: Git prevents
      developers from overwriting each other's work by only allowing changes
      that can be fast-forwarded. This means that changes must be added in a way
      that builds on the existing code, rather than replacing it.

# 21 / 27: Forking Workflows

</center>

<center>

![](data605/book/Lesson02.1-Git.png/slides021.png){width=80%}

</center>

- **Forking Workflows**
  - In many open-source projects, not all developers have the ability to make
    changes directly to the project's code. This is because only a select group
    of core contributors have read-write permissions, meaning they can make
    changes directly. Everyone else has read-only access, which means they can
    view the code but cannot make changes directly.

- **Solution**
  - The solution to this limitation is a process called "forking." Forking
    allows developers to create their own copy of the project where they can
    make changes freely.
  - **External contributors**:
    - They start by cloning the repository, which means they make a local copy
      of the project on their computer.
    - They then create a writable fork, which is essentially their own version
      of the project on the platform (like GitHub).
    - They can create branches in their fork to work on new features or bug
      fixes.
    - Once they are satisfied with their changes, they prepare a Pull Request
      (PR) to propose their changes to the original project.
  - **Project maintainer**:
    - The maintainer reviews these PRs to ensure the changes are beneficial and
      do not introduce issues.
    - If the changes are approved, the maintainer accepts and integrates them
      into the main project.
    - This process ensures that the project maintainer has control over what
      changes are made to the project.

- **Aka "GitHub workflow"**
  - This workflow is often referred to as the "GitHub workflow" because GitHub
    popularized the concept of forking with their platform. The phrase "Fork me
    on GitHub!" became a symbol of this innovation.
  - GitHub's success with this model was significant enough that Microsoft
    acquired the company for 7.5 billion USD, highlighting the value and impact
    of this approach to collaborative software development.

- **Integration-Manager Workflow**  
  This workflow is a traditional model used in open-source development. It's
  commonly seen in projects like Linux and on platforms like GitHub, where the
  concept of forking is prevalent. This model helps manage contributions from
  multiple developers efficiently.

- **One repo is the official project**  
  In this workflow, there is a single official repository that represents the
  project. The project maintainer is the only person who has the authority to
  push changes directly to this public repository. For example, in a project
  named `causify-ai/csfy`, the maintainer would be responsible for managing the
  official repo.

- **Each contributor**  
  Contributors have read access to the public repositories of others, allowing
  them to see what changes are being made. They create a personal copy of the
  project, known as a fork, where they have write access. For instance, a
  contributor might fork the project to `gpsaggese/csfy`. They make their
  changes in this fork and then push these changes to their own public
  repository. Once satisfied, they send a pull request to the maintainer,
  requesting that their changes be merged into the official project.

- **The maintainer**  
  The maintainer plays a crucial role in this workflow. They add the
  contributor's repository as a remote to their local setup, allowing them to
  pull in the changes. The maintainer then merges these changes into a local
  branch and tests them to ensure everything works correctly. Once the changes
  are verified, the maintainer pushes the updated branch to the official
  repository, integrating the contributor's work into the project. This process
  ensures that only stable and tested changes make it into the official project.

# 23 / 27: Git log

- **Git log**: The `git log` command is a powerful tool in Git that provides
  detailed information about the history of commits in a repository. It allows
  you to see what changes have been made, who made them, and when they were
  made. This is crucial for understanding the evolution of a project and for
  tracking down when specific changes were introduced.

- **Refs (References)**: In Git, references or "refs" are pointers to commits.
  They help you navigate through the history of your project:
  - **`HEAD`**: This is the current commit you are working on. It points to the
    latest commit in your current branch and is where your next commit will be
    based.
  - **`origin/master`**: This refers to the remote branch, typically the main
    branch on the remote repository. It helps you keep track of what the main
    branch looks like on the server.
  - **`experiment`**: This is an example of a local branch. Local branches are
    branches you create on your machine to work on features or fixes.
  - **`d921970`**: This is an example of a specific commit identifier (SHA-1
    hash). Each commit in Git has a unique identifier that allows you to
    reference it directly.

- **Caret (`^`) Notation**: The caret symbol is used to navigate through commit
  history:
  - **`HEAD^`**: This refers to the commit immediately before the current
    `HEAD`. It's useful for looking at the previous state of your project.
  - **`^2`**: This is used in the context of merge commits, which have more than
    one parent. `^2` refers to the second parent of a merge commit, allowing you
    to see the other branch that was merged.
  - **Merge Commits**: These are special commits that combine changes from
    different branches. They have multiple parents, which is why the caret
    notation is useful for exploring them.

# 24 / 27: Dot notation

</center>

<center>

![](data605/book/Lesson02.1-Git.png/slides024.png){width=80%}

</center>

- **Double-dot notation**
  - The double-dot notation in Git is used to compare two branches or commits.
    When you see something like `1..2`, it means you are looking at commits that
    are reachable from the second commit (or branch) but not from the first.
    Think of it as finding the "difference" between two points in your project
    history.
  - For example, if you run `git log master..experiment`, Git will show you
    commits `D` and `C`. These are the changes that exist in the `experiment`
    branch but not in the `master` branch.
  - Conversely, `git log experiment..master` will show you commits `F` and `E`,
    which are present in `master` but not in `experiment`.

- **Triple-dot notation**
  - The triple-dot notation is a bit different. When you see `1...2`, it refers
    to commits that are reachable from either of the two branches or commits,
    but not from both. It's like taking a "union" of the two sets of commits and
    then excluding the ones that are common to both.
  - For instance, `git log master...experiment` will list commits `F`, `E`, `D`,
    and `C`. These are all the commits that are unique to either `master` or
    `experiment`, excluding any that might be common to both branches.

- Advanced Git

* **Stashing**
  - _Stashing_ is a useful feature in Git that allows you to temporarily save
    changes in your working directory. This includes both modified and staged
    files. Think of it as putting your work on hold without committing it. This
    is particularly helpful when you need to switch branches or work on
    something else without losing your current progress. The changes are saved
    in a stack, which means you can have multiple stashes and apply them later
    when you're ready to continue working.

* **Cherry-picking**
  - _Cherry-picking_ is a powerful tool that lets you apply a specific commit
    from one branch to another. This is useful when you want to incorporate a
    particular change without merging entire branches. It allows for selective
    integration of changes, which can be handy in managing features or bug fixes
    across different branches.

* **rerere**
  - The term _rerere_ stands for "Reuse Recorded Resolution." This feature in
    Git helps by remembering how you resolved conflicts in the past. When
    similar conflicts arise again, Git can automatically apply the same
    resolution, saving you time and effort. It's particularly beneficial in
    projects with frequent merges and recurring conflicts.

* **Submodules / subtrees**
  - These are methods to include one Git project within another. _Submodules_
    allow you to keep a separate repository within your main project,
    maintaining its own history and independence. _Subtrees_, on the other hand,
    integrate the external project more tightly, allowing for easier management
    of changes. Both are useful for managing dependencies or incorporating
    external libraries into your project.

- **bisect**
  - _Purpose_: `git bisect` is a powerful tool used to pinpoint the exact commit
    that introduced a bug in your codebase.
    - Imagine your code is like a tree, and the bug is at the top. You know it's
      there, but you're not sure when it first appeared.
    - You start with a known good revision and a bad one (where the bug is
      present), but the exact point of introduction is unknown.
    - You can automate the process by using a script that tests each commit. The
      script should return 0 if the commit is good (bug-free) and a non-zero
      value if it's bad (bug present).
    - `git bisect` will systematically narrow down the range of commits to find
      the first bad commit, saving you time and effort.

- **filter-branch**
  - _Purpose_: This command allows you to rewrite the history of your repository
    in a flexible, scriptable manner.
    - For example, you might need to change an author's email address across all
      commits or remove a file containing sensitive information.
    - The process involves checking out each commit, applying your changes (like
      running a command), and then committing the result back into the history.
    - It's a powerful tool but should be used with caution, as rewriting history
      can affect collaborators.

- **Hooks**
  - _Purpose_: Hooks are scripts that Git can run automatically at certain
    points in your workflow, such as before committing or merging.
    - They allow you to automate tasks and enforce policies, like running tests
      before a commit or preventing certain types of merges.
    - Hooks can help maintain code quality and consistency across your team by
      ensuring that certain conditions are met before changes are accepted.
